Notas de mudança da API v2.0

ROTAS DE LOGIN E CADASTRO:

Como era antes: POST /auth/login e POST /auth/cadastro

Como ficou: POST /auth/google/login e POST /auth/google/cadastro

Motivo da Mudança: Facilitar a adição novos métodos de autenticação (ex: email e senha, facebook, twitter, etc...)

MÉTODO DE AUTENTICAÇÃO (Não muda nada no front-end):

Como era antes: UUID-Token gerado a cada login

Como ficou: JWT (Json Web Token)

Motivo da Mudança: JWT (JSON Web Token) é um método RCT 7519 padrão da indústria para realizar autenticação entre duas partes por meio de um token assinado que autentica uma requisição web: https://www.devmedia.com.br/como-o-jwt-funciona/40265

Observações: Agora a api_key não vai mudar caso o servidor seja reiniciado ou ocorra qualquer falha interna.

NOME DO API_TOKEN:

Como era antes: api_token

Como ficou: api_key

Motivo da Mudança: Padronização: https://en.wikipedia.org/wiki/Application_programming_interface_key

PADRÃO DE RESPOSTAS:

Como era antes: Não tinha

Como ficou: A resposta é baseada nos códigos HTTP. 200 é OK, e o recurso que vc quer vai estar na resposta. Qualquer outro código é um ERRO, e a resposta será um JSON com 3 atributos:

  • stack: em qual arquivo/módulo/função ocorreu o erro (apenas para fins de debug)
  • mensagem: mensagem original de erro - ex: "Cannot read property 'split' of undefined" (apenas para fins de debug)
  • mensagem_amigavel: descrição amigável do erro - (ex: usuário não cadastrado)

Observações: Os atributos stack e mensagem só aparecerão caso o NODE_ENV esteja setado para "development". Eles não existirão no modo produção (production)

Códigos de status de respostas HTTP: https://developer.mozilla.org/pt-BR/docs/Web/HTTP/Status
